~ FIREBASE AUTH SERVICE ~

	Users
	    uid: The unique, permanent ID for the user across your entire project. This is the "gold standard" key to use when saving data in your database.
	    displayName: The user's full name (e.g., "Jane Doe").
	    email: The primary email address associated with the account. (optional for anonymous users)
	    photoURL: A link to the user’s profile picture.
	    emailVerified: A boolean (true/false) indicating if the user has clicked the link in a verification email.
	    isAnonymous: true if the user is using a "Guest" account.
	    metadata.creationTime: The exact date/time the account was first created.
	    metadata.lastSignInTime: The last time the user successfully authenticated.
	    providerId: (e.g., google.com, facebook.com, password)
	    refreshToken: Used to get a new access token when the current one expires.

~ EVENTS ~

--- USER EVENTS (direct from app) ---

USER: Opens the app for the first time (no account)
	-> AUTH: anonymous user creation
	-> FIRESTORE: (handled by onAuthUserCreated trigger, see SYSTEM EVENTS)

USER: New user signup with email via Firebase Auth in app
	-> AUTH: update current anon user with email, password, displayName
	-> FIRESTORE: update Users/{uid} with profile fields (merge)

USER: User updates profile (name, photo)
	-> AUTH: update displayName, photoURL
	-> FIRESTORE: update Users/{uid} mirrored fields
	WHY DIRECT: single-document, owner-scoped write protected by security rules

USER: Edits own draft digital design metadata (name, description, tags, price)
	-> FIRESTORE: update DigitalDesigns/{digitalDesignId} (name, description, tags, priceAmount, lastUpdatedAt)
	WHY DIRECT: single-document owner write, no cross-doc side effects, design must have marketplaceStatus="Inactive"

USER: Edits own draft physical design metadata (name, description, tags, price)
	-> FIRESTORE: update PhysicalDesigns/{physicalDesignId} (name, description, tags, priceAmount, lastUpdatedAt)
	WHY DIRECT: same as above


--- USER EVENTS (via Cloud Functions) ---

USER: Uploads all assets, fills out all fields, then hits submit to create a new digital design
	-> CLOUD FUNCTION: createDigitalDesign (onCall)
		-> validates auth, fields, staged files in Storage
		-> FIRESTORE: write new document in DigitalDesigns/
		-> STORAGE: copy staged files to final folder structure in DigitalDesigns/{digitalDesignId}/
		-> STORAGE: delete temp staged files
		-> FIRESTORE: update DigitalDesigns/{digitalDesignId} with signed marketplace image URLs
	WHY FUNCTION: multi-step atomic workflow across Firestore + Storage, must not leave partial state

USER: Uploads all assets, fills out all fields, then hits submit to create a new physical design
	-> CURRENT STATE: no active cloud function implementation

USER: Publishes a digital design to the marketplace
	-> CLOUD FUNCTION: publishDigitalDesign (onCall)
		-> validates caller is authorId
		-> validates all required fields and assets exist
		-> FIRESTORE: update DigitalDesigns/{digitalDesignId} marketplaceStatus="Public", lastUpdatedAt
	WHY FUNCTION: policy enforcement, must verify asset completeness server-side

USER: Publishes a physical design to the marketplace
	-> CLOUD FUNCTION: publishPhysicalDesign (onCall)
		-> validates caller is authorId
		-> validates all required fields and assets exist
		-> FIRESTORE: update PhysicalDesigns/{physicalDesignId} marketplaceStatus="Public", lastUpdatedAt
	WHY FUNCTION: same as above

USER: Unpublishes a design from the marketplace
	-> CLOUD FUNCTION: unpublishDesign (onCall)
		-> validates caller is authorId
		-> FIRESTORE: update marketplaceStatus="Inactive", lastUpdatedAt
	WHY FUNCTION: prevents client from setting arbitrary statuses

USER: Saves a digital design from the marketplace
	-> CLOUD FUNCTION: saveDigitalDesign (onCall)
		-> FIRESTORE (transaction): update Users/{uid}.savedDigitalDesigns arrayUnion(designId)
		-> FIRESTORE (transaction): update DigitalDesigns/{digitalDesignId}.savedCount increment +1
	WHY FUNCTION: cross-document write with counter consistency requires server transaction

USER: Unsaves a digital design
	-> CLOUD FUNCTION: unsaveDigitalDesign (onCall)
		-> FIRESTORE (transaction): update Users/{uid}.savedDigitalDesigns arrayRemove(designId)
		-> FIRESTORE (transaction): update DigitalDesigns/{digitalDesignId}.savedCount decrement -1 (floor 0)
	WHY FUNCTION: same as above

USER: Saves a physical design from the marketplace
	-> CLOUD FUNCTION: savePhysicalDesign (onCall)
		-> FIRESTORE (transaction): update Users/{uid}.savedPhysicalDesigns arrayUnion(designId)
		-> FIRESTORE (transaction): update PhysicalDesigns/{physicalDesignId}.savedCount increment +1
	WHY FUNCTION: cross-document counter consistency

USER: Unsaves a physical design
	-> CLOUD FUNCTION: unsavePhysicalDesign (onCall)
		-> FIRESTORE (transaction): update Users/{uid}.savedPhysicalDesigns arrayRemove(designId)
		-> FIRESTORE (transaction): update PhysicalDesigns/{physicalDesignId}.savedCount decrement -1 (floor 0)
	WHY FUNCTION: same as above

USER: Deletes own unpublished design
	-> CLOUD FUNCTION: deleteDesign (onCall)
		-> validates caller is authorId, design marketplaceStatus != "Public"
		-> FIRESTORE: update design with isDeleted=true, deletedAt (soft delete)
		-> STORAGE: delete design folder tree (async, can be deferred)
		-> FIRESTORE: cleanup references in Users/*/savedDigitalDesigns or savedPhysicalDesigns
	WHY FUNCTION: cross-document cleanup, storage deletion, must verify ownership + status

USER: Initiates checkout / purchase
	-> CLOUD FUNCTION: createCheckoutSession (onCall)
		-> FIRESTORE: write Stripe/Customers/{uid}/CheckoutSessions/{sessionId}
	WHY FUNCTION: must create Stripe session with server-side secret key

USER: Requests account deletion
	-> CLOUD FUNCTION: deleteUserAccount (onCall)
		-> AUTH: delete user account
		-> FIRESTORE: soft-delete Users/{uid} (isDeleted=true, deletedAt)
		-> FIRESTORE: update authored designs authorDeleted=true
		-> STORAGE: schedule deletion of private user assets
	WHY FUNCTION: multi-service teardown, must be atomic and trusted


--- ADMIN EVENTS (via Cloud Functions, restricted by custom claims) ---

ADMIN: Approves a marketplace listing
	-> CLOUD FUNCTION: adminModerateDesign (onCall, requires admin custom claim)
		-> FIRESTORE: update design marketplaceStatus="Public", approvedBy, approvedAt
	WHY FUNCTION: privileged write, must verify admin role server-side

ADMIN: Rejects a marketplace listing
	-> CLOUD FUNCTION: adminModerateDesign (onCall, requires admin custom claim)
		-> FIRESTORE: update design marketplaceStatus="Inactive", rejectionReason, rejectedAt
	WHY FUNCTION: same as above

ADMIN: Removes a violating design
	-> CLOUD FUNCTION: adminModerateDesign (onCall, requires admin custom claim)
		-> FIRESTORE: soft-delete design (isDeleted=true, deletedAt, deletedReason, deletedBy)
		-> STORAGE: optionally delete public derivative assets
	WHY FUNCTION: privileged destructive action with audit trail

ADMIN: Generates garment QR codes
	-> CLOUD FUNCTION: generateGarmentQRCodes (onRequest, private) [ALREADY EXISTS]
		-> STORAGE: write Garments/{garmentId}/code_1200x1200.png
		-> FIRESTORE: update Garments/{garmentId}.qrCodeStatus="Generated"
	WHY FUNCTION: bulk batch operation, generates binary assets

ADMIN: Updates garment print/shipping status
	-> CLOUD FUNCTION: adminSetGarmentStatus (onCall, requires admin custom claim)
		-> FIRESTORE: update Garments/{garmentId}.printStatus
		-> FIRESTORE: optionally write status history subdoc
	WHY FUNCTION: privileged lifecycle update with audit trail

ADMIN: Uploads shipping label for garment
	-> CLOUD FUNCTION: adminUploadShippingLabel (onCall, requires admin custom claim)
		-> STORAGE: write Garments/{garmentId}/ShippingLabels/{filename}.pdf
		-> FIRESTORE: update Garments/{garmentId} with shippingLabelUrl, carrier, trackingNumber
	WHY FUNCTION: privileged write, cross-service (Storage + Firestore)

ADMIN: Assigns garment owner manually
	-> CLOUD FUNCTION: adminAssignGarmentOwner (onCall, requires admin custom claim)
		-> FIRESTORE: update Garments/{garmentId}.owner = targetUid
		-> FIRESTORE: update Users/{targetUid}.ownedGarments arrayUnion(garmentId)
	WHY FUNCTION: cross-document privileged write

ADMIN: Runs field migration
	-> CURRENT STATE: no active cloud function implementation


--- SYSTEM / TRIGGER EVENTS (automated, no user action) ---

SYSTEM: Firebase Auth user created (trigger)
	-> CLOUD FUNCTION: onAuthUserCreated (onAuthUserCreated trigger)
		-> FIRESTORE: write baseline Users/{uid} document (set with merge, idempotent)
	WHY FUNCTION: guaranteed user doc creation without relying on client

SYSTEM: Stripe webhook checkout.session.completed
	-> CLOUD FUNCTION: stripeWebhookHandler (onRequest)
		-> validates Stripe signature with webhook secret
		-> FIRESTORE: write Stripe/Customers/{uid}/Payments/{paymentId}
		-> FIRESTORE: update design purchaseCount increment +1
		-> FIRESTORE: write Garments/{garmentId} if purchase creates order item
		-> FIRESTORE: update Users/{uid}.ownedGarments arrayUnion(garmentId)
	WHY FUNCTION: must verify Stripe signature, uses server secret, cross-document writes

SYSTEM: Stripe webhook customer.subscription.updated / deleted
	-> CLOUD FUNCTION: stripeWebhookHandler (onRequest)
		-> FIRESTORE: update Stripe/Customers/{uid}/Subscriptions/{subId}
		-> AUTH: optionally update custom claims / role
	WHY FUNCTION: same as above

SYSTEM: Scheduled cleanup of orphaned temp uploads (e.g. daily)
	-> CLOUD FUNCTION: scheduledCleanupOrphans (onSchedule, every 24h)
		-> STORAGE: delete temp staged files older than 24h
		-> FIRESTORE: remove dangling references in saved arrays pointing to deleted designs
	WHY FUNCTION: maintenance task, no user trigger, cost control
